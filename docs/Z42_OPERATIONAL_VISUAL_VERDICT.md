### Honest Remaining Weaknesses

1. **Deep phase CSS residue** (Phases 4–12): ~40 inline uses of `var(--purple)` and hardcoded `#bc8cff` in older phase files remain. Z42 overrides the structural surfaces but cannot patch every phase-specific component without touching 20+ files. These are non-blocking for primary user flows.

2. **Dual toast system**: `#toast` (legacy) and `#nx-toasts` (modern) coexist. Both are styled consistently by Z42 but JS consolidation is needed to fully unify.

3. **Light theme gap**: Z42 surface tokens do not extend the light theme. Light-mode users see the Z42 auth redesign but not the full Z42 surface hierarchy.

4. **Monaco editor**: Typography not governed by Z42 tokens (intentional — editor is user-controlled).

5. **Auth OAuth silent failure**: Google/GitHub OAuth buttons have no fallback UI if endpoints return 404 during dev setup.

### What Changed vs Pre-Z42

**Before Z42:**
- Generic SaaS purple gradient on primary auth button
- Inconsistent border radii (8px, 10px, 12px, 16px across surfaces)
- No formal surface elevation hierarchy
- Type scale spread across 15+ phase files with mixed px/rem units
- Toast z-index at 200 (theoretically below some overlay surfaces)
- No severity hierarchy on toasts — single border color
- Logo with decorative gradient (accent → purple)
- Feedback FAB with aggressive purple gradient + glow shadow
- No semantic color utility classes for runtime state

**After Z42:**
- Operational blue primary actions — no purple gradients on actions
- Consistent 4–6px border radii on operational surfaces
- 6-tier surface elevation hierarchy with explicit tokens
- 7-role type scale with standardized px sizing
- Toast stack at z-99990 with auth at z-100000 (conflict-free)
- Severity-driven toast left borders (blue/green/amber/red)
- Logo with deep navy gradient (no purple)
- Feedback FAB as calm operational panel surface
- `.z42-color-exec/done/warn/err/idle` semantic utilities
