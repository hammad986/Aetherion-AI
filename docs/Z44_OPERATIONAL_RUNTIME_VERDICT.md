### Honest Remaining Weaknesses

1. **DAG ↔ Timeline hover sync**: Architecture ready, JS bridge not wired (requires touching Z30 DAG). Estimated effort: 20–30 lines of JS.

2. **`paused`/`stabilizing`/`queued` backend signals**: These states fire only when the backend status text matches specific patterns. If the backend uses different status words, these states won't activate. A backend status enum normalization would make this bulletproof.

3. **Replay cursor sync**: Z33 timeline scrub does not emit events that Z44 can intercept. The replay state CSS is ready but the cursor position sync requires Z33 modification.

4. **Mission strip on historical log load**: When `loadOlderLogs()` replays old entries into `#logArea`, those entries don't flow through the live `MutationObserver` path. The mission strip only updates for live-appended log lines.

5. **Advisory action buttons**: The advisory messages are text-only. Injecting contextual action buttons (Retry, Resume, Inject) would enhance the operator guidance UX but requires wiring to existing JS action handlers.

6. **Session age JS timing**: Session age is calculated from when Z44 detects `RUNNING` state, not from the actual session `started_at` timestamp. For precise age, the JS should read `started_at` from the status API response — available but not yet wired.
